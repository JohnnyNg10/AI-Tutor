# AI Tutor 评测体系设计

> 目标：构建可量化、可复现、可归因的产品评测体系，发现 Bad Case 并驱动迭代

---

## 一、评测维度框架（6 维度 × 22 指标）

### 1.1 推荐质量维度

衡量 Advisor Agent 推题是否"恰到好处"——既不太难也不太易。

| 指标 | 定义 | 计算方式 | 目标值 | Bad Case 信号 |
|------|------|---------|--------|--------------|
| 难度匹配度 | 推荐题目难度与用户θ值的偏差 | `｜diff - θ_hat｜` 的滑动平均 | < 0.8 | 连续 5 题偏差 > 1.5 → 难度失配告警 |
| 知识点覆盖率 | 推荐题目覆盖用户薄弱知识点的比例 | `推荐命中薄弱KP数 / 用户薄弱KP总数` | > 60% | 薄弱点从未被推荐 → 长尾忽略 |
| 推荐重复率 | 相似题目(同KP+同难度)在窗口内的重复比例 | `窗口内重复题数 / 窗口大小` | < 15% | 连续 3 题同KP同难度 → 多样性不足 |
| 冷启动首推准确率 | 新用户(无历史)第1题推荐后答对的比例 | `新用户首题正确数 / 新用户总数` | > 45% | < 30% → 冷启动策略偏难 |
| 探索/复习平衡度 | 新题与复习题比例偏离目标(7:3)的程度 | `｜新题占比 - 0.7｜` | < 0.15 | 连续 10 题全是新题 → 复习队列失效 |

### 1.2 教学效果维度

衡量 Instructor Agent 的分步讲解是否真的"教会了"学生。

| 指标 | 定义 | 计算方式 | 目标值 | Bad Case 信号 |
|------|------|---------|--------|--------------|
| 提示有效率 | 查看提示后下一次提交的正确率 | `hint_dismiss 后答对数 / hint_dismiss 总数` | > 55% | < 35% → 提示内容不解决问题 |
| 提示递进合规率 | 严格按L0→L1→L2→L3→L4递进的比例 | `合规递进链数 / 总提示链数` | > 50% | < 20% → 提示层级设计不合理，学生跳过中间层 |
| 讲解后正确率 | 查看完整方案后同类题正确率 | `看方案后同类题答对数 / 同类题总数` | > 70% | < 45% → 讲解未有效传递知识点 |
| 苏格拉底对话轮次 | Instructor 引导学生自主得出结论的平均对话轮次 | `avg(每道题的引导对话轮次)` | 3-6 轮 | < 2 轮 → 太直接给答案；> 10 轮 → 过于绕弯 |

### 1.3 认知诊断维度

衡量 BKT + IRT + Soft Label 对用户能力的刻画是否准确。

| 指标 | 定义 | 计算方式 | 目标值 | Bad Case 信号 |
|------|------|---------|--------|--------------|
| BKT 收敛速度 | 掌握度估计达到稳定所需的答题次数 | `P(L) 连续3题波动 < 0.05 时的答题数` | < 8 题 | > 15 题 → 收敛太慢，冷启动体验差 |
| θ 估计稳定性 | IRT 能力值在答对/答错时的波动幅度 | `｜θ_new - θ_old｜` 的分布 | 单题波动 < 0.3 | 答对一题 θ 跳变 > 0.6 → 估计不稳定 |
| 软标签预测力 | 六维软标签对后续答题正确率的预测力 | `soft_label 与未来 10 题正确率的 Spearman ρ` | ρ > 0.3 | ρ < 0.1 → 软标签和实际表现脱节 |
| 掌握度假阳性 | 标记为"精通"但同类题答错的比例 | `精通节点答错数 / 精通节点答题总数` | < 10% | > 20% → 掌握度阈值偏宽松 |

### 1.4 用户体验维度

衡量产品流畅度和可用性。

| 指标 | 定义 | 计算方式 | 目标值 | Bad Case 信号 |
|------|------|---------|--------|--------------|
| 首次响应延迟 | 用户提交到 SSE 首字出字的耗时 | `P50 / P95 / P99` | P50 < 1.5s, P95 < 4s | P95 > 6s → 用户感知到卡顿 |
| 会话完成率 | 进入会话到有效产出(至少完成1题或收到1条有效回复)的比例 | `有效会话数 / 总会话数` | > 85% | < 70% → 用户频繁中断，可能是体验问题 |
| 错误率 | 接口 4xx/5xx 占比 | `错误响应数 / 总请求数` | < 2% | 特定端点错误率突增 → 功能异常 |
| 推荐页加载时长 | RecommendView 从进入到首题渲染的时间 | `P50 / P95` | P50 < 2s | P50 > 4s → Redis 或 ChromaDB 性能问题 |

### 1.5 个性化维度

衡量自适应策略是否真正"千人千面"。

| 指标 | 定义 | 计算方式 | 目标值 | Bad Case 信号 |
|------|------|---------|--------|--------------|
| 模式切换合理性 | Advisor 模式切换后学生正确率的变化方向 | `切换后10题正确率 - 切换前10题正确率` | 切换后提升 > 0 | 切换到CHALLENGE后正确率暴跌 → 切换过早 |
| 难度递进平滑度 | 推荐难度的逐题变化幅度 | `｜diff_t - diff_{t-1}｜` 的滑动平均 | < 1.2 | 连续 4 题难度单调递增 → 可能超出学生能力 |
| 学习轨迹改善率 | 使用 2 周后 θ 值提升的用户比例 | `θ_{2周} > θ_{初始} 的用户数 / 总用户数` | > 70% | < 50% → 产品未产生学习效果 |
| 退课预警 | 连续 3 天未登录的用户比例 | `连续3天未登录用户数 / 活跃用户总数` | < 20% | > 30% → 用户留存危机，需排查体验问题 |

### 1.6 鲁棒性维度

衡量系统在边界和异常场景下的表现。

| 指标 | 定义 | 计算方式 | 目标值 | Bad Case 信号 |
|------|------|---------|--------|--------------|
| 冷启动可用性 | 新用户(无任何历史)能否完成首次推荐 | `新用户首推成功率` | > 95% | < 90% → 冷启动链路有阻塞点 |
| 恶意输入过滤率 | 对乱码/超长/无意义输入的正确处理率 | `正确拦截或优雅降级数 / 异常输入总数` | > 95% | 输入 "asdfghjkl" 返回正常题目 → 未过滤 |
| 服务降级覆盖率 | Redis/ChromaDB 不可用时功能降级成功率 | `降级成功数 / 服务故障测试数` | > 90% | 某个服务挂掉直接白屏 → 降级链路未覆盖 |
| 知识图谱健壮性 | 缺失知识点时的兜底表现 | `缺失KP的题目被正确处理(不崩溃)的比例` | 100% | 新KP导致崩溃 → Schema 扩展性不足 |

## 二、评测集构建

### 2.1 标准化用户画像（5 类）

基于 IRT θ 值构建模拟用户，覆盖从新手到专家的全能力区间：

| 画像 | θ 初始值 | 已掌握 KP 数 | 薄弱 KP 数 | 学习风格 | 模拟目标 |
|------|---------|-------------|-----------|---------|---------|
| P1-数列新手 | -2.0 | 0/83 | 15+ | 频繁求助 | 验证冷启动 + SCAFFOLD 模式 |
| P2-基础薄弱 | -0.5 | 15/83 | 8 | 跳题倾向 | 验证薄弱点定位 + 坚持性软标签 |
| P3-中等水平 | +0.5 | 40/83 | 4 | 正常 | 验证 ENCOURAGE 模式 + 推荐多样性 |
| P4-进阶高手 | +1.5 | 65/83 | 2 | 独立解题 | 验证 CHALLENGE 模式 + 迁移能力 |
| P5-学霸选手 | +2.5 | 78/83 | 0 | 快速答题 | 验证天花板表现 + 复习策略 |

### 2.2 模拟答题轨迹（每个画像 4 条轨迹）

为每个画像设计 4 条模拟答题轨迹，每条轨迹 30 题，覆盖不同的行为模式：

| 轨迹 | 行为特征 | 测试目的 |
|------|---------|---------|
| T1-正常规律 | 答对率 65%，渐进式求助，认真看解析 | 基准线 |
| T2-焦虑模式 | 答对率 40%，频繁 L3/L4 求助，快速跳过，短期放弃 | 坚持性/自主性低场景 |
| T3-超常发挥 | 答对率 85%，几乎不求助，仔细读解析 | 能力增长与模式切换 |
| T4-波动模式 | 答对率在 30%-80% 间剧烈波动，求助层级跳跃 | θ 估计稳定性 |

### 2.3 边界与异常用例

| 用例 | 触发条件 | 预期行为 |
|------|---------|---------|
| E1-空白输入 | 提交答案为空字符串 | 前端拦截，不发送请求 |
| E2-超长输入 | 提交 10000+ 字符的答案 | 截断处理，正常返回 |
| E3-极速答题 | 1 秒内完成审题+作答 | 标记为异常，不计入 θ 更新 |
| E4-连续错误 | 连续 15 题答错 | 触发降级保护模式，难度下限锁定 |
| E5-零交互 | 打开应用后 10 分钟无操作 | session_end 正常上报，无内存泄漏 |
| E6-并发请求 | 同一用户 5 个并发答题请求 | 乐观锁保护，无数据写脏 |
| E7-Redis 宕机 | 停掉 Redis 后答题 | 回退 MySQL 直读，功能不中断 |
| E8-模型超时 | LLM 调用超时(>30s) | 重试 1 次 → 返回兜底回复 |

### 2.4 评测数据准备

| 数据 | 来源 | 数量 |
|------|------|------|
| 题库 | knowledge_base/question/ | 130+ 题 |
| 知识点树 | knowledge_graph.py | 83 节点 |
| 答题记录 | user_interaction_logs (积累后) | 真实用户数据 |
| 参考答案 | 每题的 answer 字段 | 130 条 |
| 难度标注 | 每题 difficulty 字段 | 130 条 |

## 三、指标分级与看板设计

### 3.1 三级指标体系

```
L1 北极星指标（1-2个）
├── 学习轨迹改善率：使用后 θ 提升的用户比例
└── 周活跃留存率：本周活跃用户中上周也活跃的比例

L2 核心质量指标（6个，对应6个维度各1个最关键的）
├── 难度匹配度          → 推荐质量
├── 提示有效率           → 教学效果
├── BKT收敛速度          → 认知诊断
├── 首次响应延迟 P95     → 用户体验
├── 模式切换合理性       → 个性化
└── 冷启动可用性         → 鲁棒性

L3 诊断指标（其余14个）
└── 用于发现具体 Bad Case 和定位根因
```

### 3.2 Bad Case 自动发现规则

```
IF 难度匹配度 > 1.5 持续 5 题 → [推荐-难度失配] 生成 Case
IF 提示有效率 < 0.35 持续 10 题 → [教学-提示失效] 生成 Case
IF 掌握度假阳性 > 0.20 → [诊断-虚假精通] 生成 Case
IF P95 延迟 > 6s 持续 30min → [体验-性能劣化] 生成 Case
IF 模式切换后10题正确率下降 > 0.2 → [个性化-切换失误] 生成 Case
IF 冷启动首推成功率 < 90% → [鲁棒性-冷启动阻塞] 生成 Case
```

### 3.3 Bad Case 分析模板

```markdown
## Case #{id}: {简短标题}

**发现时间**: {timestamp}
**触发规则**: {rule_name}
**严重程度**: P0/P1/P2

### 现象
- 用户画像: {profile_type}
- 触发场景: {scenario}
- 异常数据: {关键指标偏离值}

### 根因定位
1. 数据层: {是否有数据缺失/脏数据}
2. 算法层: {BKT/IRT/推荐算法是否有问题}
3. 交互层: {前端是否触发了预期外的行为}
4. 模型层: {LLM 输出是否不符合预期}

### 影响范围
- 影响用户类型: {画像}
- 影响频率: {每N次触发1次}

### 改进方案
- 短期止血: {hotfix}
- 中期修复: {算法/策略调整}
- 长期根治: {架构/设计变更}

### 验证方式
- 复现条件: {最小复现步骤}
- 修复验证指标: {指标需恢复到什么水平}
```

## 四、评测执行流程

```
              ┌──────────────────────┐
              │  1. 模拟用户答题     │
              │  (30题 × 5画像 ×     │
              │   4轨迹 = 600次)     │
              └────────┬─────────────┘
                       ▼
              ┌──────────────────────┐
              │  2. 指标自动计算     │
              │  (22 个指标全量输出) │
              └────────┬─────────────┘
                       ▼
              ┌──────────────────────┐
              │  3. Bad Case 自动发现│
              │  (8 条规则自动扫描)  │
              └────────┬─────────────┘
                       ▼
              ┌──────────────────────┐
              │  4. 人工归因分析     │
              │  (Bad Case 模板填写) │
              └────────┬─────────────┘
                       ▼
              ┌──────────────────────┐
              │  5. 改进方案评审     │
              │  (P0 立即修 / P1 本周│
              │   / P2 下迭代)       │
              └────────┬─────────────┘
                       ▼
              ┌──────────────────────┐
              │  6. 回归验证         │
              │  (修复后重新跑评测)  │
              └──────────────────────┘
```

## 五、与竞品的对比评测维度

面试时如果被问到"和市面上其他AI教育产品比怎么样"，可以用以下框架回答：

| 对比维度 | 传统题库App(作业帮等) | 通用AI chatbot(ChatGPT) | AI Tutor |
|---------|---------------------|------------------------|----------|
| 推荐逻辑 | 固定难度递进 | 无个性化追踪 | BKT+IRT双轨 + 软标签联合判定 |
| 反馈粒度 | 二元对错 | 直接给答案 | L0-L4渐进提示 + Actual Score四因子 |
| 知识追踪 | 无 | 无记忆 | 83节点逐个BKT + 艾宾浩斯衰减 |
| 教学模式 | 无差别展示答案 | 单轮问答 | 三模式自适应 + 苏格拉底引导 |
| 可解释性 | 无 | 无 | 六维画像 + 知识树 + 能力曲线 |
| 长尾知识点 | 冷门知识点无题可刷 | 可能编造错误答案 | KG约束 + 题库审核 + ChromaDB语义召回 |

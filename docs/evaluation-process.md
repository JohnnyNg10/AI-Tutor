## 二、测试用例设计

### 2.1 用户画像（5 类）

为覆盖从新手到学霸的完整用户光谱，定义了 5 类标准化画像：

| 画像 | θ 初始值 | 掌握的 KP | 薄弱 KP | 求助偏好 | 跳题倾向 |
|------|---------|----------|---------|---------|---------|
| P1-数列新手 | -2.0 | 0/83 | 15+ | 频繁求助 | 0.2 |
| P2-基础薄弱 | -0.5 | 15/83 | 8 | 正常 | 0.6 |
| P3-中等水平 | +0.5 | 40/83 | 4 | 正常 | 0.15 |
| P4-进阶高手 | +1.5 | 65/83 | 2 | 独立 | 0.05 |
| P5-学霸选手 | +2.5 | 78/83 | 0 | 独立 | 0.02 |

### 2.2 行为轨迹（4 种）

每个画像搭配 4 种不同的答题行为模式，模拟真实场景中的行为差异：

| 轨迹 | 行为特征 | 答对率 | 测试目的 |
|------|---------|--------|---------|
| T1-正常规律 | 渐进式求助 L0→L1→L2，认真看解析 | ~65% | 基准线 |
| T2-焦虑模式 | 频繁 L3/L4 求助，快速跳过，短期放弃 | ~40% | 坚持性/自主性低场景 |
| T3-超常发挥 | 几乎不求助，仔细读解析 | ~85% | 能力增长与模式切换 |
| T4-波动模式 | 正确率 30%-80% 剧烈波动，求助层级跳跃 | 波动 | θ 估计稳定性 |

### 2.3 评测矩阵（20 条模拟链路）

```
        T1       T2       T3       T4
P1  [新手×正常] [新手×焦虑] [新手×超常] [新手×波动]
P2  [薄弱×正常] [薄弱×焦虑] [薄弱×超常] [薄弱×波动]
P3  [中等×正常] [中等×焦虑] [中等×超常] [中等×波动]
P4  [高手×正常] [高手×焦虑] [高手×超常] [高手×波动]
P5  [学霸×正常] [学霸×焦虑] [学霸×超常] [学霸×波动]
```

每条链路模拟 30 道题的完整答题流程：获取推荐 → 答题 → 提交反馈。

**总计：5 × 4 × 30 = 600 次 API 调用**。

## 三、答题行为模拟机制

### 3.1 答对/答错概率计算

使用简化的 IRT 2PL 模型估算每个模拟学生在每道题上的正确概率：

```
P(correct) = 1 / (1 + exp(-a * (theta - b)))

其中：
  a = 1.0  （区分度参数）
  θ = 用户的当前能力值（由画像决定）
  b = 题目难度 - 2.0（映射到 θ 尺度）
```

不同轨迹在此基础上引入行为偏差：
- **T2 焦虑模式**：P(correct) × 0.6（焦虑拉低正确率）
- **T3 超常发挥**：P(correct) × 1.3（上限 95%）

### 3.2 求助行为模拟

```
P1 正常规律：
→ 40% 概率使用提示 → 按 L0→L1→L2 渐进式递进

P2 焦虑模式：
→ 70% 概率使用提示 → 跳跃式，直接选 L3/L4

P4/P5 独立型：
→ 10% 概率使用提示 → 仅用 L0/L1
```

### 3.3 跳题行为模拟

```
跳题概率 = 画像的 skip_tendency 参数 × 当前状态因子
  P2 焦虑模式在低潮期(phase=0)：额外 +30% 跳题概率
  跳题原因：too_hard(70%) / too_easy(20%) / tired(10%)
```

### 3.4 每次答题的完整 API 调用链

```
1. GET  /advisor/recommend?limit=5     → 获取 5 道推荐题目
2. 取推荐列表的第 1 道题
3. POST /advisor/feedback               → 提交答题反馈（答对/答错/提示次数/耗时/跳题原因）
4. 每 5 题额外调用 GET /advisor/profile → 获取画像快照（记录 θ 和 mastery 变化）
```

## 四、执行过程

### 4.1 三轮迭代调试

**第 1 轮 — 数据流断裂**

```
现象：20 条轨迹全部跑完（OK），但指标全部 unknown
排查：检查保存的 JSON 结果文件
发现：feedbacks_count = 0，_raw_recommendations 为空
根因：
  ① submit_feedback() 在 API 返回空 data 时 return None，上层 if fb: 判断为 False
  ② API 返回的推荐数据 key 是 recommendations 而非 questions，匹配失败
修复：
  ① 改为始终返回包含 is_correct 的字典
  ② 同时检查 recommendations / questions 两个字段名
```

**第 2 轮 — 部分指标仍为 unknown**

```
现象：会话完成率 = 0.0，提示有效率 = 0.0
排查：检查 _raw_feedbacks 为空
根因：to_dict() 在序列化时重新构建 feedbacks 字典，但 theta/mastery 取值逻辑有误
修复：
  ① to_dict() 增加 _raw_recommendations、_raw_actions、_raw_feedbacks 完整导出
  ② metrics_engine 增加 _extract_question_items() 统一解析多种 API 格式
```

**第 3 轮 — GBK 编码导致报告生成中断**

```
现象：完整跑完 20 条轨迹后，报告生成阶段崩溃
错误：UnicodeEncodeError: 'gbk' codec can't encode character '⚠'
根因：Windows 控制台默认 GBK 编码无法输出 emoji 符号
修复：sys.stdout.reconfigure(encoding="utf-8", errors="replace")
```

## 五、结果分析

### 5.1 全局结果

| 维度 | 健康率 | 状态 | 主要问题 |
|------|--------|------|---------|
| 用户体验 | 100% | 健康 | 会话完成率 100%，API 无超时/崩溃 |
| 个性化 | 72% | 危险 | 模式切换后正确率下降 13-27% |
| 推荐质量 | 40% | 警告 | 知识点覆盖率仅 50%，存在重复推题 |
| 认知诊断 | 35% | 危险 | θ 跳变高达 1.1（目标 <0.3） |
| 教学效果 | 30% | 危险 | 提示有效率仅 0-21%（P1/P2） |

### 5.2 按画像维度的关键发现

```
P1 数列新手：难度匹配偏差最大(1.11)，θ 跳变最大(1.02)
  → 冷启动阶段的推荐质量是最大痛点

P2 基础薄弱：焦虑模式下 7 次 API 422 错误（题目 ID 不存在）
  → 题库中存在失效题目 ID  

P3 中等水平：波动模式下后半段正确率下降 27%（模式切换过早）
  → ENCOURAGE→CHALLENGE 切换阈值需调高

P4 进阶高手：所有指标健康，θ 稳定性好(0.16)
  → 老用户体验最佳，说明算法在数据充足时表现良好

P5 学霸选手：知识点覆盖率仍为 50%（无薄弱点可推荐导致随机推题）
  → 学霸场景缺少"拔高题"和综合题
```

### 5.3 按轨迹维度的关键发现

```
T1 正常规律：基准线表现最好，提示有效率随画像能力提升而提升
T2 焦虑模式：提示有效率最低(0-9%)，错误率最高
T3 超常发挥：θ 跳变最小(0.16-0.31)，算法在稳定输入下表现良好  
T4 波动模式：θ 跳变最大(0.88-1.02)，算法对不稳定输入敏感
```

## 六、本次评测的局限性

| 局限 | 说明 | 后续改进 |
|------|------|---------|
| 模拟用户非真实用户 | IRT 2PL 简化模型无法完全模拟真实学生行为 | 积累真实用户数据后回归验证 |
| 未测试 LLM 质量 | Instructor Agent 的回复质量未纳入指标 | 引入 LLM-as-Judge 评估提示/讲解质量 |
| 未测试鲁棒性 | Redis 宕机、模型超时等边界场景未跑 | 后续增加混沌工程测试用例 |
| 题目难度为模拟值 | 未使用题库中真实难度标注 | 接入 knowledge_base 真实题目数据 |

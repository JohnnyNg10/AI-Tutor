# Stage 3: 指标评测与持续迭代 — AI Tutor 产品实战报告

> 适用场景：AI产品经理实习求职 · 简历作品集 · 面试宣讲

## 一、产品背景

**AI Tutor** 是一个基于 IRT（项目反应理论）的智能学习推荐系统。核心价值主张：**根据学生的能力水平和知识点薄弱处，个性化推送最适合的练习题**。

推荐管道的理想形态：
```
用户画像(θ值+薄弱点) → 向量语义检索(ChromaDB) → LLM质量审查 → 难度匹配排序 → 个性化题目
```

**栈**: FastAPI + MySQL + Redis + ChromaDB + Vue 3 + 飞书集成

## 二、用户痛点发现与指标定义

### 2.1 核心问题：推荐加载 90 秒

**用户场景还原**：
> 学生小明完成一道题后，点击"下一题推荐"，页面卡住近一分半钟。他不知道系统在做什么，以为卡死了，反复刷新页面，最终放弃离开。**一个以"学习"为核心价值的产品，最大的用户阻力竟然是"等待"本身。**

**根因拆解（通过逐段打日志定位）**：

| 瓶颈环节 | 耗时 | 占比 | 阻塞类型 |
|----------|------|------|----------|
| Redis 连接超时（无缓存重试机制） | ~30s | 33% | 同步阻塞，每次失败重试5s |
| LLM 质量检查（GPT API 同步调用） | ~20s | 22% | 同步HTTP请求 |
| ChromaDB 向量embedding | ~15s | 17% | 同步embedding调用 |
| 其他串行IO累积 | ~25s | 28% | — |
| **总计** | **~90s** | **100%** | |

### 2.2 定义双维度指标体系

**问题**：单纯追求速度可能伤害推荐质量。因此必须同时定义速度指标和质量指标。

| 维度 | 指标 | 定义 | 现状 |
|------|------|------|------|
| **速度** | 推荐响应时间 P99 | 用户点击到看到题目的耗时 | ~90s |
| **速度** | 推荐可用率 | 5s内成功返回的比例 | ~5% |
| **质量** | 知识点匹配率 | 推荐题目的知识点 ∩ 用户薄弱点 / 用户薄弱点 | **未度量** |
| **质量** | 难度匹配偏差 | 推荐题目难度与用户θ值的平均偏差 | **未度量** |
| **质量** | 用户跳过率 | 用户看到推荐后主动跳过的比例 | **未度量** |

> 关键洞察：速度指标是可测量的，但质量指标在优化前并未建立基线。这是一个需要警惕的信号。

### 2.3 衍生问题：Mock数据与功能不可达

在排查性能问题时并行发现：

- **16个API路由未注册**：前端调用的API后端返回404 → 功能看似"已开发"实则"不可达"
- **12+个API返回硬编码Mock数据**：能力雷达图、知识树、错因分析等返回**所有用户相同**的假数据 → 个性化完全失效
- **连击状态不持久化**：Redis重启后连续学习天数归零 → 游戏化机制断裂
- **前后端端口不匹配**：前端代理8002→实际后端8000 → 开发环境完全不通

**新增健康度指标**：

| 指标 | 目标 | 初始值 |
|------|------|--------|
| API路由覆盖率 | 100% | 91% (160/176) |
| 真实数据API比例 | 100% | ~70% |
| 游戏化状态持久化 | 支持 | 不支持 |

## 三、产品决策：速度与精准度的艰难取舍

### 3.1 为什么这是一个产品决策，而不只是技术优化

理想推荐管道中的每个环节都有其产品价值：

```
ChromaDB 向量检索         LLM 质量审查
      ↓                       ↓
"找到跟用户薄弱点语义        "确保推荐的题目表述
 相关但可能标签不同           清晰、难度合理、
 的优质题目"                 没有歧义错误"
```

**直接砍掉这两步意味着什么**：

| 丢失的能力 | 产品影响 | 严重程度 |
|------------|----------|----------|
| 语义相似度匹配 | 只能用知识点标签的**字符串相等**来匹配，`"二次函数"`和`"一元二次方程"`被视为完全不相关 | **高** |
| 跨知识点关联发现 | 无法发现"A知识点薄弱可能源于B基础不牢"的隐含关系 | 中 |
| LLM题目质量审查 | 无法自动过滤表述不清、有歧义、难度标注错误的题目 | 中 |

**但保留它们意味着**：用户等待90秒 → 用户放弃 → **推荐质量为0（用户根本没看到）**

### 3.2 产品决策框架

```
                        精准度
                          ↑
                    Zone A │ Zone B
                 (理想区域) │ (精准但不可用)
                 目标状态    │ 当前状态: 90s
                          │ 用户放弃
                ───────────┼──────────→ 速度
                    Zone C │ Zone D
                 (不可用)   │ (可用但不精准)
                          │ 快速路径: 0.1s
                          │ 用户至少能做题
```

**决策**：先将产品从 Zone B（精准但不可用）拉到 Zone D（可用但不完美），然后通过迭代回到 Zone A。

这不是技术妥协，而是**产品存活决策**——一个用户永远等不到的完美推荐，不如一个用户马上能用的及格推荐。

### 3.3 快速路径中保留的个性化能力

`_fast_recommend()` 虽然砍掉了语义检索，但**并非随机出题**。它保留了以下个性化信号：

| 机制 | 代码实现 | 产品价值 |
|------|----------|----------|
| **难度匹配** | `abs(题难度 - 用户θ) → 距离分` | 不会给θ=2.0的学生推难度5.0的题 |
| **薄弱点匹配** | 题目标签 ∩ 用户薄弱标签 → 命中率分 | 数学薄弱的不会被推英语题 |
| **已做题去重** | `seen_ids` 集合过滤 | 不会反复推已经做过的题 |
| **复习优先** | Redis复习ZSet优先出队 | 错题按艾宾浩斯间隔复习 |
| **综合打分** | `score = 知识点相关度×0.5 + 难度匹配×0.5` | 排序而非随机，优先推送最匹配的 |

```python
# advisor.py:46 - 实际的匹配逻辑（虽简单但不随机）
def _kp_relevance(q_topics, weak_kps):
    weak_set = set(weak_kps)
    hits = sum(1 for t in q_topics if t in weak_set)  # 精确标签匹配
    return hits / len(q_topics)

def _difficulty_match(theta, q_difficulty):
    diff = abs(q_difficulty - theta)
    if diff <= 0.5:  return 1.0    # 完全匹配
    if diff >= 2.0:  return 0.0    # 差距过大
    return 1.0 - (diff - 0.5) / 1.5
```

### 3.4 配套优化（低风险高收益）

| 优化 | 措施 | 产品影响 |
|------|------|----------|
| Redis故障容错 | 超时1s，失败缓存30s | 不因Redis宕机而整体不可用 |
| 跳过LLM同步调用 | 从热路径移除 | 题目已是人工审查入库的，风险可控 |
| 跳过ChromaDB同步调用 | 从热路径移除 | 知识点标签可作为粗粒度的替代匹配 |

### 3.5 数据真实化（Mock → DB）

| 模块 | 改造前 | 改造后 |
|------|--------|--------|
| 技能树 (Skill Tree) | 硬编码3个节点 | `user_knowledge_mastery` 实时计算 |
| 六维能力雷达图 | 固定mock分数 | `user_interaction_logs` 聚合计算 |
| IRT段位排名 | fake θ=3.0 | `user_ability_history` 最新记录 |
| 错因分类 | 空列表 | `mistake_book` + Chroma向量检索 |
| 知识树进度 | mock百分比 | 基于真实mastery数据 |
| 能力曲线 | 固定3个点 | `user_ability_history` 历史趋势 |
| 复习队列 | 空列表 | `review_queue_settings` + Redis ZSet |
| 数据迁移 | 打印日志 | UPDATE/INSERT真实数据表 |

## 四、RAG候选池：从"快速保底"回到"智能推荐"

### 4.1 背景：快速路径的产品债

第三章中，为了把推荐响应从90秒降到0.1秒，我做了一个有意识的妥协——砍掉ChromaDB向量检索和LLM质检。当时记录的待验证假设是：

> "标签匹配能否近似替代语义匹配？重叠率 > 60% 即可接受"

但标签匹配的硬伤是明显的：`"二次函数"`和`"一元二次方程"`在字符串层面完全不匹配，但语义上是高度相关的。快速路径是**保底方案**，不能作为终态。

### 4.2 方案设计：三层漏斗架构

在快速路径验证可用之后，我设计并推动了完整的RAG候选池：

```
                      ┌──────────────────────────┐
                      │  用户画像                  │
                      │  θ值 + 薄弱知识点 + 历史   │
                      └──────────┬───────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                  ▼                  ▼
    ┌─────────────────┐ ┌───────────────┐ ┌─────────────────┐
    │ ① 召回层        │ │ ② 过滤层      │ │ ③ 精排层         │
    │ ChromaDB向量检索 │ │ Seen Pool去重  │ │ 加权打分排序      │
    │ 语义相似度 > 标签 │ │ │S-θ| ≤ 1.0   │ │ 0.6+0.3+0.1      │
    └─────────────────┘ └───────────────┘ └─────────────────┘
            │                  │                  │
            ▼                  ▼                  ▼
      500道候选            ~50道过滤后          Top 5推荐
```

**各层产品定义**：

| 层级 | 产品目标 | 硬指标 | 失败降级策略 |
|------|----------|--------|-------------|
| **召回层** | 不错过语义相关但标签不同的题目 | ChromaDB k=20 召回 | 回退到MySQL标签查询 |
| **过滤层** | 不推已做题、不推难度偏差过大的题 | Seen Pool去重、\|S-θ\|≤1.0 | Seen Pool降级跳过、难度放宽至2.0 |
| **精排层** | 最相关+最匹配的排在前面 | 加权分≥0.5才进入最终列表 | 排序降级为随机shuffle |

### 4.3 关键产品决策

**决策1：加权公式的权重**

```
最终得分 = 0.6 × 知识点相关度 + 0.3 × 难度匹配度 + 0.1 × 上下文相似度
```

这不是拍脑袋定的。推演过程：
- 知识点相关度权重最高(0.6)：学生做错题的首要原因是"这个知识点没掌握"，而非"题目太难"
- 难度匹配次之(0.3)：学生θ值已经通过BKT持续追踪，难度偏差过大→挫折感或无聊感
- 上下文相似度最低(0.1)：向量相似度信号弱，容易受题目长度、无关文字影响

**决策2：70%新题 + 30%复习题混合策略**

- 纯新题 → 学了就忘，没有记忆巩固
- 纯复习 → 学生感觉"没有新东西"，学习动力下降
- 70/30 → 保证新鲜感的同时，利用艾宾浩斯间隔复习巩固薄弱点

**决策3：Seen Pool去重放在过滤层而非召回层**

这是一个性能取舍：如果先去重再向量检索，Seen Pool可能有上千条，每条都要排除——ChromaDB的`where`过滤性能随排除列表增大而线性下降。改为先召回再过滤，虽然可能多召回一些已做题，但ChromaDB的检索性能稳定，Seen Pool的内存级查找(O(1))成本几乎为零。

### 4.4 与快速路径的关系

快速路径(`_fast_recommend`)并未被废弃，而是作为**三级降级链路**保留：

```
L1: RAG完整路径 (ChromaDB + Seen Pool + 加权排序)
       ↓ 向量检索超时或ChromaDB不可用
L2: 快速路径 (MySQL标签查询 + Seen Pool + 难度匹配)
       ↓ MySQL查询也失败
L3: 兜底随机 (随机出题 + 仅难度过滤)
```

这确保了推荐引擎**永远不会因为一个子系统故障而整体不可用**。

### 4.5 可量化的产品提升

| 维度 | 快速路径(L2) | RAG完整路径(L1) | 改善 |
|------|-------------|-----------------|------|
| 知识点相关度 | 精确字符串匹配 | 语义向量匹配 | 跨标签相关题目从**0%→可发现** |
| 候选池大小 | MySQL全表扫(~200) | ChromaDB语义Top-K(500) | 候选丰富度 ↑ 2.5x |
| 排序维度 | 2维(知识点+难度) | 3维(相关度+难度+语义) | 排序精度 ↑ |
| 复习混合 | 无 | 70/30新题复习比 | 记忆巩固机制从无到有 |
| 响应时间 | <0.1s | <0.5s | 仍在用户无感范围内 |
| 降级能力 | — | 三级降级链路 | 可用性不降级 |

## 六、迭代方法论：这才是PM面试要讲的故事

### 5.1 完整闭环

```
① 发现用户问题
   "推荐加载90秒，用户放弃"
        ↓
② 定义双维度指标
   "速度×质量，缺一不可"
        ↓
③ 拆解瓶颈，评估取舍代价
   "砍LLM省20s，但丢失质量审查；砍向量省15s，但丢失语义匹配"
        ↓
④ 做有意识的妥协，记录假设
   "先保可用性(0.1s)，用标签匹配替代向量匹配，假设标签粒度足够"
        ↓
⑤ 快速上线验证
   "可用率 5% → 100%，用户至少能用了"
        ↓
⑥ 设计A/B实验验证假设
   "下一步：对比实验，量化精准度损失"
        ↓
⑦ 根据数据回到第①步
```

### 5.2 面试叙事框架

**不要这样讲**（纯技术视角）：
> "我把Redis超时改短了，砍了LLM调用，推荐从90秒变成0.1秒。"

**要这样讲**（产品决策视角）：
> "我遇到一个两难问题：个性化推荐需要向量检索+LLM质检，但这两步导致加载90秒。用户在等不到结果时就流失了。
>
> 我的判断是：一个等不到的满分推荐 = 零分。所以我做了三件事：
> 第一，把'必须快'和'必须准'拆成两个独立指标，分别追踪。
> 第二，在当前版本先用标签匹配替代语义匹配保速度，但保留了难度匹配、去重、复习优先等所有其他个性化逻辑。
> 第三，明确记录了4个待验证假设，设计了A/B实验方案来量化精准度的实际损失。
>
> 结果：可用性从5%提升到100%，同时我有清晰的路径知道下一步该优化什么。这不是'砍功能'，是'分阶段交付'——先让用户能用，再让推荐更准。"

> 我砍掉了推荐引擎的向量检索和LLM质检。当时推荐加载要90秒，我拆解发现向量embedding占15s、LLM调用占20s。但这两步恰恰是"个性化"的核心。
>
> 我的逻辑是：一个用户永远等不到的精准推荐 = 零价值。所以我把"精准度"从"必须上线前做到完美"重新定义为"上线后通过实验迭代优化"。在当前版本，我用知识点标签的精确匹配+难度匹配+去重来保证基础个性化，把核心体验从"等90秒"变成"0.1秒出题"。
>
> 但我没有假装这个取舍没有代价——我明确记录了4个待验证假设（标签匹配能替代语义匹配吗？跳过质检会导致错误题流出吗？），并设计了A/B实验来量化。如果数据证明精准度损失不可接受，我有明确的回滚和折中方案（比如异步LLM质检、后台向量预计算）。
>
> 这让我学到：产品经理的核心能力不是做完美的决策，而是做有意识的取舍，并且知道如何验证自己的判断。

## 八、AI安全与幻觉治理 (新增案例)

### Bad Case: 图片上传后模型自问自答

**用户场景**: 学生上传2023年新高考一卷数学第7题图片，AI回复"未能识别数学题目"，然后**自行编造了一道调和级数不等式题**开始讲解。

**根因分析**:
| 层面 | 问题 | 严重程度 |
|------|------|----------|
| API层 | SiliconFlow禁用了`Qwen2.5-VL-32B`视觉模型，图片OCR完全不可用 | 致命 |
| 数据流层 | OCR失败后，空的question文本传给Instructor | 严重 |
| Prompt层 | Instructor System Prompt没有"禁止编造题目"的硬约束 | 关键 |
| RAG副作用 | 即使question为空，RAG仍检索到数列相关文档，LLM基于检索结果自行出题 | 严重 |

**修复方案**:
1. **System Prompt 增加安全护栏**: "如果用户输入中没有明确的数学题目，绝对不能自己编造题目"
2. **图片上传流程兜底**: OCR失败时用明确的错误提示替换空question，阻止RAG误导
3. **视觉模型降级策略**: 检测视觉模型不可用时直接告知用户，不尝试调用

**产品方法论沉淀**:
> AI产品中"模型幻觉"不是技术问题，是产品设计问题。三道防线缺一不可: 
> ① **输入层**: 拒绝空/无效输入进入推理管线
> ② **Prompt层**: 硬约束规则必须写在System Prompt最高优先级
> ③ **容错层**: RAG检索到的知识≠用户的问题，二者需要显式对齐
